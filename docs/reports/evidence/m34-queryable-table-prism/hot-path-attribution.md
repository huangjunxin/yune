# M34 hot-path attribution

Date: 2026-06-23

## Attribution method

M34 used the fresh native full-ABI versus engine-only rows, the cross-engine
`RimeProcessKey` plus context harness, a full reader audit of translator and
engine code, and one temporary diagnostic run before choosing levers. The
temporary diagnostic was generated by `cargo test` with a gated test-only helper
and is archived as `hot-path-attribution-diagnostic.txt`; the helper was removed
before final verification, so no diagnostic instrumentation is retained in
production code.

The strongest attribution signal is the gap between full-ABI and engine-only
`luna_pinyin` rows:

| Row | Before median | After median | Change |
| --- | ---: | ---: | ---: |
| `ni` full ABI | `1,760.250 us` | `1,132.950 us` | `-35.6%` |
| `ni` engine-only | `569.700 us` | `575.250 us` | `+1.0%` |
| `zhongguo` full ABI | `12,697.600 us` | `12,119.013 us` | `-4.6%` |
| `zhongguo` engine-only | `532.575 us` | `515.713 us` | `-3.2%` |

This says the largest accepted M34 win is not raw map lookup speed. It is
avoiding expensive first-page ABI/context work caused by carrying a full
materialized candidate vector through `Engine::refresh_candidates` and
`RimeGetContext` when only a page is needed.

Diagnostic counts: `ni` had `182` exact candidates, `8` completion codes,
`417` completion candidates, `295` eligible/eager candidates, and `20` bounded
candidates. `hao` had `139` exact candidates, no completion codes, `77`
eligible/eager candidates, and `20` bounded candidates. The debug diagnostic is
not treated as the release benchmark; its role is to prove raw index and prefix
enumeration are tiny compared with materialization and context/page emission.
Filters, rankers, userdb merge, and AI merge were disabled or absent for these
measured rows; context/menu cost is represented by the fresh full-ABI versus
engine-only benchmark gap above.

## Owner split

| Owner | Finding | M34 action |
| --- | --- | --- |
| Lookup spec generation | Happens before table lookup; not the accepted dominant owner for `ni`. | Left unchanged. |
| Exact index access | `entries_by_code.get(...)` is cheap relative to full-list materialization. | Wrapped by `TableLookup` for later storage swaps. |
| Completion range scan | Short prefixes enumerate many code rows; under code-order storage, early stop is unsafe. | Kept complete enumeration; bounded only expensive materialization. |
| Abbreviation/comment formatting | Existing eager path clones candidates, probes spelling-abbreviation metadata, and formats comments before paging. | Bounded path keeps candidate references, quality, and emission order until top-K is selected. |
| Engine global sort/filter | Full-list `sort_by` and filter application require fully materialized candidates. | Bounded path is enabled only when no full-list ranker/userdb/filter semantics require eager fallback. |
| Context/menu update | `Engine::snapshot()` clones the context candidate vector before `RimeGetContext` emits one page. | Lazy candidate window reduces the vector for first-page `luna_pinyin` rows. |
| Query-time spelling algebra | Not present as the `ni`/`hao` query-time owner; current schema install builds expansions before typing. | No per-key win is claimed from spelling algebra. |
| Storage representation | Heap map is still used. Cold/memory remain storage work. | Deferred to later queryable table/prism work. |

## Decision

Lever A ran first because full-result materialization and context/menu work were
the safe owner. Storage rewrites stopped at the lookup abstraction gate because
current compiled readers are owned parsers, the prism does not carry candidate
payloads, and dynamic correction/TypeDuck behavior still need precise
fallbacks before deleting or bypassing `entries_by_code`.

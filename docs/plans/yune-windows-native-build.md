# Yune Windows Native Build

> **Status:** Parked - **Milestone:** M10 (TypeDuck-Windows compatibility profile) - **Updated:** 2026-06-19 - **Type:** reference

This is parked TypeDuck-Windows native package reference material. After M12,
Yune's default `RimeApi` follows upstream `rime/librime 1.17.0` and does not
expose TypeDuck fork-only `config_list_append_*` slots. The package script now
fails fast until a named TypeDuck profile ABI surface exists.

A future TypeDuck profile package should produce the weasel-shaped layout:

```text
dist/
  include/
    rime_api.h
    rime_levers_api.h
  lib/
    rime.dll
    rime.lib
    rime.pdb        # present when Cargo emits it
```

## Prerequisites

- Rust target: `x86_64-pc-windows-msvc`
- MSVC linker/toolchain available to Cargo
- TypeDuck fork headers from the v1.1.2 oracle archive, defaulting to:

```powershell
target\typeduck-oracle\v1.1.2\extract\dist\include
```

Those headers are used because `RimeApi` field order is the C ABI. Historical
Item 2 notes verified TypeDuck fork slot order after `config_list_size` and
before `get_input`. After M12, those fork-only slots are not in the default
upstream table; any future TypeDuck profile ABI must re-derive its slot order
from the fork header. In the v1.1.2 header, `config_begin_list` is earlier in
the table, next to `config_begin_map`.

## Obtaining The v1.1.2 Oracle

The oracle is external state, not generated by Yune. A clean checkout does not
contain the TypeDuck fork binaries, headers, or compiled schema build output.
When present locally, the files live under the gitignored directory:

```text
target\typeduck-oracle\v1.1.2\
```

The pinned provenance is:

- `TypeDuck-HK/librime` (<https://github.com/TypeDuck-HK/librime>), tag `v1.1.2`,
  commit `74cb52b78fb2411137a7643f6c8bc6517acfde69`
- `TypeDuck-HK/schema` (<https://github.com/TypeDuck-HK/schema>), commit
  `1bed1ae6a0ab48055f073774d7dfd152a171c548`
- `TypeDuck-HK/rime-dictionary-lookup-filter`
  (<https://github.com/TypeDuck-HK/rime-dictionary-lookup-filter>), commit
  `3e4605c4fae99f068df2edb85aaeab5a97752795`

There is no checked-in fetch script yet. To reproduce the local oracle state,
obtain or build the fork at the pinned revision, extract its `dist\include`
headers to `target\typeduck-oracle\v1.1.2\extract\dist\include`, and build the
pinned schema with the fork's `rime_deployer.exe` into the local
`target\typeduck-oracle\v1.1.2\rime-user\build` tree. Expected bytes must come
from that fork output, never from Yune.

## Build And Package

Current behavior from the repository root:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\package-typeduck-windows.ps1
```

The script fails fast with the M12 parked/profile message. Do not use it as a
current package or smoke gate.

Archived pre-M12 behavior ran:

```powershell
cargo build -p yune-rime-api --release --target x86_64-pc-windows-msvc
```

Then it copied:

- `target\x86_64-pc-windows-msvc\release\yune_rime_api.dll` to `dist\lib\rime.dll`
- `target\x86_64-pc-windows-msvc\release\yune_rime_api.dll.lib` to `dist\lib\rime.lib`
- `rime_api.h` and `rime_levers_api.h` to `dist\include\`

Default output:

```text
target\typeduck-windows-native\x86_64-pc-windows-msvc\dist
```

Use `-OutputDir` or `-HeaderSource` only after the script is re-enabled for a
named TypeDuck profile ABI surface.

## Smoke Check

Smoke checking is parked. The old smoke loaded `dist\lib\rime.dll`, resolved
`rime_get_api`, verified a positive `data_size`, and read a TypeDuck fork-only
`config_list_append_string` slot. That fixed-slot check is not valid against the
default upstream `RimeApi` after M12.

A future TypeDuck profile smoke must call the named profile API and re-derive
the slot offset from TypeDuck-HK/librime `v1.1.2` `rime_api.h`. Do not skip the
future smoke except on a host that cannot load the Windows DLL:

```powershell
powershell -ExecutionPolicy Bypass -File scripts\package-typeduck-windows.ps1 -SkipSmoke
```

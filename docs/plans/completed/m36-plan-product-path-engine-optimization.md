This plan is ONE roadmap milestone composed of independently-shippable, byte-parity-gated, stop-gated phases. It is explicitly NOT a big-bang: phases may land separately. **Goal:** Materially narrow (not close) the librime gap on the schema users actually run — TypeDuck `jyut6ping3` — by profiling first, then fixing the verified product-path waste in safe Rust. Lead with measurement; concentrate effort on the few dominant cost owners; keep `my_rime`/librime as references, never oracles or code sources; keep public ABI and TypeDuck profile behavior byte-identical.

**Closeout:** M36 landed the native in-process Track A/Track B benchmark harness, recorded the shipped TypeDuck product blobs as stale/unsupported for Yune (`marisa string_table`, unsupported prism, unsupported reverse), closed `rsmarisa` by measured no-go for this milestone, and selected schema-scoped no-marisa re-emitted compiled assets as the passing product path. Deployment now writes configured product prism stems such as `jyut6ping3_mobile`, and product compact storage is enabled only when compiled artifacts load. Track B `jyut6ping3_mobile` final rows run with `compiled_ready=true`, product typing rows improve up to `81.7%`, and product max peak working set drops from `1000.4 MB` to `885.3 MB`. Track A `luna_pinyin` remains comparison-only and still trails librime widely. Browser/runtime/Cloudflare/OpenCC delivery work remains M31.

**Why this milestone is framed differently from M33–M35:** M33–M35 each moved the needle only slightly because they optimized the **comparison path (`luna_pinyin`)**, while the **product path (`jyut6ping3`)** never engaged the new machinery. Verified in-tree:

- Compact storage is gated **off** for `jyut6ping3` (`crates/yune-rime-api/src/schema_install.rs:325`, `!is_typeduck_jyut6ping3_profile`).
- The parsed prism payload is **discarded** on the product's heap path: `StaticTableTranslator::from_dictionary` sets `prism_payload: None` (`crates/yune-core/src/translator/mod.rs:350`); only `from_compact_dictionary` accepts it (`:384`).
- M34's bounded/lazy fast path is hard-gated to `luna_pinyin` with input `<= 2` chars (`crates/yune-core/src/engine.rs:1230`), so it **literally never fires for the product**.
- **[corrected 2026-06-24, thanks to GPT review]** The product `jyut6ping3.table.bin` (and `jyut6ping3_scolar`) **do** carry a marisa `string_table` (verified: `u32@60`→ self-relative offset to `We love Marisa.`, `u32@64` size = 488,504). Yune rejects any marisa table (`compiled_table.rs:285`) and **falls back to parsing the 3.6 MB source `jyut6ping3.dict.yaml` into the heap** (`schema_install.rs:1147-1156`, `SourceFallback`). So marisa rejection is **product-critical** — it forces the product onto the slow source-YAML heap path. The fix is a product compiled-table strategy: first try the existing pure-Rust [`rsmarisa`](https://docs.rs/rsmarisa/latest/rsmarisa/) reader for the embedded marisa string table. This is not a random compatibility bet: `rsmarisa` is documented as a port of original marisa-trie `0.3.1`, and the local librime oracle checkout uses `deps/marisa-trie/VERSION = 0.3.1`. Fall back to re-emitting a Yune-native no-marisa table only if `rsmarisa` fails real-blob byte-parity, WASM, MSRV, or performance gates. Only the reduced `luna_pinyin` reverse-dep and `cangjie5` are marisa-free (verified `0/0`).
- The `UpstreamSentenceModel` (a near-full second heap copy of the dictionary, `crates/yune-core/src/poet/mod.rs:161-199`) is attached **only** for `is_upstream_luna_pinyin_profile` (`schema_install.rs:362-364`) — it is a **comparison/test-harness** memory owner, **never** built for the shipped `jyut6ping3` product.

**Honest ceiling (conditional on Phase 0 profiling):** With ALL phases done, product `jyut6ping3` memory plausibly reaches ~1.3–2x an equivalent librime browser build; the fair `luna_pinyin` comparison reaches ~2–3x. This **narrows, it does not close** the gap unless measurement proves otherwise. The previous ceiling assumed no reusable marisa reader; `rsmarisa` may improve the ceiling if it can consume the embedded string table byte-identically and cheaply. The irreducible residual is structural: native mmap can help native apps, but WASM has no OS demand-paging; `rsmarisa::Trie::mmap()` is documented as unavailable on WASM, so browser work must use byte-backed mapping/loading (`Trie::map` or an equivalent owned/borrowed bytes path) plus lazy lookup evidence. Per [`yune-vs-librime-gap-not-win`], this is gap-narrowing until the data proves a real close. **The numeric ceiling above is a hypothesis until Phase 0 identifies the real product owner; do not pre-commit it in public copy.**

**Tech Stack:** Rust (`yune-core`, `yune-rime-api`), `StaticTableTranslator`, `Engine`, `CandidateRequest`/`TranslationResult`, `dictionary::{compiled_table, compiled_prism, double_array}`, `poet`, `schema_install`, `frontend_baselines`, `rsmarisa` marisa-trie reader/mapping spike (`rsmarisa` is marisa `0.3.1`-compatible, matching librime's vendored marisa), dhat-rs + massif/heaptrack, browser heap probes, upstream `luna_pinyin` + TypeDuck `jyut6ping3` fixtures, TypeDuck-Web runtime/browser gates only when runtime/browser-visible files change.

## Measurement Model — Two Distinct Tracks (read this first)

M36 keeps **two measurement tracks** that answer different questions and must never be conflated. Conflating them is the root cause of three underwhelming milestones.

- **Track A — fair cross-engine comparison (Yune vs librime).** Schema: **`luna_pinyin` only** — the shared, apples-to-apples schema both engines support. Answers "how does Yune's engine stack up against librime?" `jyut6ping3` is deliberately **excluded** from this track (there is no clean librime + TypeDuck-profile baseline). A win here is tagged **COMPARISON** and is never presented as a user-facing gain.
- **Track B — product optimization (Yune before vs after).** Schema: **`jyut6ping3`** — what TypeDuck-Web users actually type. There is **no librime baseline**; the comparison is Yune-against-itself across the change. This is the deliverable: making what ships faster/smaller. Measured both **native** and **in-browser**.

**Both tracks run on a native in-process harness — not a managed `.NET`/P‑Invoke host.** The existing cross-engine runner (`scripts/yune-vs-librime-benchmark.cs` → `YuneVsLibrimeBenchmark.exe`) hosts both engines' C ABI inside a .NET process, which contaminates exactly the numbers we care about: its CLR/GC working set is counted as Yune's RSS (inflating the ~182 MB figure), and per-call P/Invoke marshalling + `GetProcessMemoryInfo` sampling dominate and destabilize per-key latency (~60% run-to-run swings — M35's "−38%" was noise). The fix is a small **native harness** (Rust calling Yune directly; librime via its C ABI; steady-state RSS sampling) with **no managed runtime in the measured process**. Linux/Ubuntu is preferred for profiling (massif/heaptrack/perf) and cleaner latency, **but the OS is not the fix — dropping the managed host is**; note on Linux librime's real mmap demand-paging may make its RSS look even smaller, which is honest, not a Yune regression.

Why `jyut6ping3` is in this plan at all: M33–M35 optimized and measured Track A (`luna_pinyin`) while users run Track B (`jyut6ping3`), and the new machinery never engaged on the product path — so each milestone produced an honest comparison number that helped zero users. M36 keeps Track A strictly fair *and* makes Track B the optimization target.

## Scope

In scope:

- Profile-first attribution of the real dominant memory/latency/startup owners on **both** the product (`jyut6ping3`) and comparison (`luna_pinyin`) paths, native and WASM, before any optimization.
- A trustworthy **native in-process** measurement harness replacing the managed `.NET`/P‑Invoke cross-engine runner (which counts CLR/GC RSS as Yune and is ~60% noisy), keeping Track A (`luna_pinyin` vs librime) and Track B (`jyut6ping3` Yune before/after) separate and tagged.
- Free build-flag/asset wins: right-size the WASM `INITIAL_MEMORY` (128 MB up-front reservation, `scripts/typeduck-wasm-build.sh:299`) and gate the comparison-only `luna_pinyin` assets out of the product bundle.
- String interning / byte-arena on the **product storage path**: first on the current Heap/source-YAML fallback if that is still the measured owner, then preserved when Phase 2.0 moves the product to compact compiled storage.
- **A product compiled-table format Yune can actually read** (the product `jyut6ping3` table is marisa and currently rejected → source-YAML fallback): preferred = use `rsmarisa` to read/map the embedded marisa string table byte-identically; fallback = re-emit a Yune-native no-marisa compact table into the product bundle at build time.
- Canonical-and-unexpanded product table via the prism: stop discarding the prism payload during product translator initialization and stop eagerly expanding spelling algebra into owned entries. Compact compiled storage is preferred; a heap fallback is acceptable only when recorded as a measured Phase 2.0 no-go.
- Sharing/collapsing the `UpstreamSentenceModel` duplicate (comparison-only, sized only if Phase 0 confirms it dominates).
- Generalizing M34's bounded/lazy pipeline past the `luna_pinyin`/`<=2`-char gate to the product, with bounded top-K, bounded filter/ranker, and bounded context snapshot.

Out of scope / deferred:

- **native mmap without lazy borrowed lookup** — mapping bytes and then rebuilding owned heap maps is not a win. Native `rsmarisa::Trie::mmap()` is in scope only when the query path directly uses the mapped trie/string table and records file-lifetime/rebuild-invalidaton behavior. WASM uses `Trie::map`/bytes-backed loading or an equivalent path because `mmap()` is unavailable there.
- **custom marisa reader from scratch** — out of scope unless `rsmarisa` fails a required product gate and the user explicitly accepts a research-grade fallback. The product first tries `rsmarisa`; re-emitted no-marisa assets are the safe fallback path.
- Changing candidate order, comments, userdb learning, TypeDuck profile invariants, or `RimeApi`/`RimeCandidate` ABI to make performance easier.
- Treating `my_rime`/`typeduck.hk` as behavior oracles or copying their source.
- M31 Cloudflare/PWA/delivery work; M32 AI product work; Windows TSF work.

## Highest-Leverage Strategies (where the effort goes)

1. **Profile + fix measurement first** (P0). The root cause of three underwhelming milestones was mis-targeting, not weak effort.
2. **Right-size WASM `INITIAL_MEMORY` + gate comparison-only assets out of the product** (P0). A self-inflicted slice of the gap; near-free.
3. **String interning / byte arena on the product storage path** (P1). The legal analog to marisa dedup; biggest memory win available without mmap/marisa. It may start on the Heap fallback but must carry through to the compact product path after P2.0.
4. **Get the product onto a Yune-readable compiled table, then store it canonical-unexpanded via prism** (P2). Today the product `jyut6ping3` table is marisa → rejected → source-YAML heap fallback; first try `rsmarisa` against the embedded marisa string table, with native mmap and WASM byte-backed loading measured separately. If it fails, re-emit a Yune-native no-marisa table. Then stop discarding the prism payload and stop expanding spelling algebra into owned entries. The headline product structural win — librime's actual design.
5. **Collapse the `UpstreamSentenceModel` duplicate** (P2, COMPARISON-ONLY — size only if P0 confirms it dominates the `luna_pinyin` resident).
6. **Bounded/lazy pipeline generalized to the product** (P3). The lever that finally moves product per-key latency (M34's only fired for `luna_pinyin`).

## Explicitly Deprioritized (long tail — do NOT spend the milestone here)

Bit-packed/quantized weights; varint offsets; minimal-perfect-hash; front-coding; standalone DAWG; `Cow`/`SmolStr` micro-churn beyond the arena; SIMD/`target-cpu=native`/BOLT/PGO (startup is allocation-bound, not compute-bound; PGO/native-cpu are also comparison-fairness flags); custom WASM allocator (Emscripten malloc conflict); `-Oz`/streaming-instantiation (module code is tiny vs the data working set). These are recorded so they are not re-proposed; each is marginal next to the six above.

## Risk Register (from the completeness critic)

- **Dictionary pruning is a silent-quality landmine.** If build-time per-code candidate capping is ever attempted, it is gated on FIRST expanding the parity corpus to cover the pruned tail — no current fixture catches a dropped rare-but-correct candidate. (Not in the six highest-leverage items; listed to prevent accidental adoption.)
- **userdb/session growth** is a resident owner that grows AFTER the P0 snapshot; the `INITIAL_MEMORY` sizing must budget for a long-session userdb HashMap, not just the static dictionary image.
- **marisa is product-relevant after all** (the product `jyut6ping3`/`_scolar` tables are marisa → rejected → source-YAML fallback). The product fix starts with **`rsmarisa`** (Phase 2.0); re-emitted Yune-native no-marisa tables are the fallback if `rsmarisa` fails byte-parity, MSRV/WASM, or performance gates. Don't let a custom marisa reader block the product — use `rsmarisa` first, then re-emit if needed.

## Appendix A — Marisa string_table read support (deferred optional path)

**Corrected premise:** marisa is **not** comparison-only — the shipped product `jyut6ping3`/`jyut6ping3_scolar` tables carry a marisa `string_table` and are rejected at `compiled_table.rs:285`, forcing the source-YAML fallback. So the product needs *a* readable compiled table. Phase 2.0 first evaluates [`rsmarisa`](https://docs.rs/rsmarisa/latest/rsmarisa/), a pure-Rust marisa-trie port that documents lookup, reverse lookup, common-prefix search, predictive search, `mmap()` for native, and `map()` for static byte slices. `rsmarisa` is documented as a port of marisa-trie `0.3.1`, matching the local librime oracle checkout's vendored `deps/marisa-trie/VERSION`. This is now the preferred marisa path because it avoids writing a custom LOUDS reader, but it must still prove byte-identical reads against the actual product blobs, including any nested/multi-trie structure.

Re-emitting a Yune-native no-marisa table remains the fallback, not the first choice. The fallback is still fair and useful because each engine may build its own best deployed representation from the same source dictionary. A **custom** marisa reader from scratch is out of scope unless `rsmarisa` fails and the user explicitly accepts that research risk.

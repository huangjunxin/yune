<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| local schema YAML -> schema installation | Schema YAML may name arbitrary components and namespaces; component recognition must not execute plugins or arbitrary code. |
| schema component name -> diagnostic/deferral record | Component strings cross from untrusted config into internal diagnostics and must be stored as data, not executed. |
| schema resource ID -> runtime filesystem lookup | Any compatibility increment that reads data must preserve logical resource-ID validation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Elevation of Privilege | remaining gear recognition | mitigate | Recognition of `poet`, `grammar`, `contextual_translation`, and plugins must never load or execute external plugin code; unsupported plugin behavior becomes structured deferral. |
| T-03-02-02 | Tampering | `schema_install.rs` component parsing | mitigate | Reuse `schema_component_prescription` and validation helpers; never build filesystem paths or dynamic symbol names directly from raw component strings. |
| T-03-02-03 | Spoofing | compatibility diagnostics | mitigate | Deferral records must use explicit target phases and must not present unsupported gears as compatible implementations. |
| T-03-02-04 | Denial of Service | large schema component lists | accept | Schema installation already processes local deployed YAML lists; this plan adds constant-time recognition/records per listed component and no unbounded external lookup. |
</threat_model>
